The gap between “interesting arxiv paper” and “code I can actually use” is 1-4 weeks. We automated it.

PaperForge is a pipeline that takes an arxiv paper ID and produces production-ready code deployed on AWS Bedrock, multi-model benchmark results, and a structured technical report. Average time: 2-3 hours. Average cost: $5-15 in API calls.

The reproduction crisis is a deployment crisis

Over 60% of ML papers can’t be reproduced from published artifacts. But even when code exists, it’s rarely production-ready:

  • Research code uses OpenAI or HuggingFace APIs → production needs Bedrock
  • No retry logic, no authentication, no cost tracking
  • Single-model evaluation → you need to know which Bedrock model works best
  • No quality gates → how do you know the reproduction is correct?

PaperForge solves all four problems in a single automated pipeline.

How it works

The pipeline has 8 stages, orchestrated by an AI coordinator agent:

1. Paper Analysis → Download arxiv PDF, extract methodology, datasets, claimed results

2. Code Discovery → Find associated GitHub repos, clone, inventory dependencies

3. Code Understanding → LLM reads the entire codebase and generates a structured summary

4. Cloud API Refactoring → Replace research APIs with Bedrock Converse API using an adapter pattern

5. Multi-Model Evaluation → Run experiments across multiple Bedrock models (fair comparison on identical infrastructure)

6. Quality Gate → Automated checks: code safety, architecture review, result verification

7. Report Generation → Structured technical report with methodology, results, and cost analysis

8. Publication → Push to GitHub with architecture diagrams

Details of the pipeline orchestration and quality gate methodology are the subject of ongoing work.

The paper scoring system

Not every paper is worth reproducing. PaperForge includes a pre-pipeline scoring system that evaluates:

  • Relevance: How well does this align with your focus areas?
  • Code availability: Is there a GitHub repo? How maintained is it?
  • Trending: Citation velocity, social media attention
  • Reproduction difficulty: S/M/L/XL based on dependencies and compute
  • Customer potential: Can this be applied to real use cases?

Papers scoring above threshold automatically enter the pipeline queue.

What we learned

1. Code summarization is the highest-leverage step. Having an LLM read the entire codebase before refactoring dramatically reduces errors. The CODE_SUMMARY.md it produces becomes the blueprint for everything downstream.

2. Multi-model comparison drives adoption. Customers don’t want to know that “Claude works.” They want to know that “Claude Sonnet is 20% cheaper than Opus with 5% lower accuracy on this specific task.” PaperForge provides this automatically.

3. Quality gates prevent garbage. Without automated review, ~30% of reproductions have issues (wrong metrics, missing edge cases, stale API calls). The quality gate catches these before they reach the report.

Open source

PaperForge is available at github.com/JunjieTang-D1/research-papers. We plan to release the pipeline framework as a standalone tool in Q2 2026.

Pipeline: Claude Opus 4.6 via AWS Bedrock. Orchestration: Atlas + Claude Code (ACP). Multi-model evaluation: Claude Sonnet, Nova Pro, DeepSeek V3, GLM 4.7, Kimi K2.5.
